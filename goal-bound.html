GoalBound Case Study

About GoalBound

GoalBound (formerly "10 Letters") is a web application designed to help CSUN's baseball team optimize their performance through weekly goal-setting.

CSUN Baseball Team's Coach, Gregory Moore, developed a method to promote strength and versatility by modeling and committing to goals and improving skills daily and weekly.

GoalBound is a digitized version of Gregory Moore's method, which allows individuals to build self-control, analyze their goals, and improve via cohesive goal-setting and evaluation.

The evaluation is done through self-assessment based on an honor system, which reinforces personal accountability.

My Role

My role in this project was to research, design, and test user interfaces related to the project. I was a mentor and the lead UX designer on an Agile team comprised of five student developers, a designer, a product owner, a scrum master, and four other staff mentors.

The Goalbound application was already heavily under development when Coach Gregory Moore shared details of how he would like the ability to assign mentors to mentees. Since we were working in an Agile scrum environment, I had to be responsive to changing requirements while working with what was already built.

I learned the requirements from our product owner and collaborated with the lead front-end developer to come up with a good user experience workflow. Since this was my first main UX task on this project, I was eager to get started.

The Design Process

These are the steps that I took to solve this design problem.

1. Defining the Problem

Before I could make any progress with this application, I took time to familiarize myself with all the documentation on file, including the client proposal and the paper version of Coach Greg Moore’s goal-setting methodology.

2. Explore

Ideation

I discovered that the coach view within the application lacked a clear user flow. Our goal became to rethink the way the coach interacted with and assigned roles within his team.

The sticky notes allowed my colleague and me to mentally run through each user flow to easily identify missing details and roadblocks and iterate solutions.

We created user stories based on the list of task items and then created several whiteboard wireframes of what each component might look like. Drawing rough wireframe sketches helped us think and talk through each possible flow and served as a springboard to communicate and visualize our proposed solutions.

This wireframe would eventually serve as a user testing tool.

Wireframing & Prototyping

I created wireframe screens in Adobe XD to ensure that the task flow was present for each user story.

After building the initial wireframe, I turned each screen into a clickable prototype.

As I wireframe, I organize and clearly label where each user flow starts. Organizing in this way makes it easier to understand the context of each flow when collaborating with other teammates and presenting to clients.

3. Testing

Preparing User Questions

My colleague and I prepared a list of all the actions Coach Moore should perform while managing his team roster and tested out the prototype with each other before bringing in external users (students who have never seen or used the application before).

(I think it’s always good to sanity-check your user tests before bringing in users.)

User Testing

We were finally ready to collect real user feedback with a series of in-person user tests at this phase. I asked users to perform each task while thinking aloud so that we could get a good idea of what sections were confusing or pleasant to use.

I took note of any moments I saw the user hesitate with their cursor. Part of observing a user test is looking for patterns in behavior, so when multiple users got stuck in the same spot, it was a clear sign that something in the user experience needed modification.

My colleague and I alternated administering the test while the other would be taking notes and filming. Being mentors, we felt it was important to record our user tests to review anything we may have missed and educate the UI/UX students on how a user test is run.

Iterating on User Feedback

As with any user test, we collected feedback and modified our wireframes accordingly.

We began the second round of user tests with the modifications that resulted in a much more “intuitive” user flow.

After this second round of testing, we made small modifications to our wireframe again and then passed on the wireframes to the development student teams to start building and the design student to begin working on a high fidelity mockup.

Because our team worked in Agile sprints and I wanted to accommodate enough time for our design student to work, we compromised by having the development team build the application based on wireframes initially and styling would be applied afterward.

4. Design

UI Style Tile

To keep a cohesive design direction, I encouraged our design student to create a style tile that incorporated CSUN's Brand Identity.

UI Mockup

Since we planned for the CSUN's baseball team to use the application, we applied  CSUN branding to the wireframe to create a high-fidelity mockup. Here I allowed our design student to work on the mockup while I helped give feedback and guidance.

5. Results

Client Demo

We demonstrated the UI mockup to our client, Coach Moore. He was pleased with our work and approved of the flow and design. After gaining approval, we passed off the high-fidelity mockups to the front-end development team.

With the latest update, the coach was able to manage his team as well as assign peer mentors. The end result was a clear UI that helped encompass multiple use cases.

Challenges

Joining Late

I joined the team after Goalbound had already started and begun development. Ideally, user experience design would precede development, but I saw the situation as a growth opportunity. I know that I won't always have the luxury of joining a project from the very beginning.

Limited access to the user

A challenge for this project was having limited access to testing with Coach Moore. He had a very busy schedule and was notorious for missing client demos. Knowing this, we had to improvise user testing.

One way we handled this challenge was to do in-house testing. The students tested the user flow described in this case study. I also had students test other parts of the application where the baseball team was expected to set goals within the application.

Although our student developers were not part of a baseball team, they were still close enough to the target user demographic that in-house testing was better than none at all.

What I learned

Because of this project, I realized how important it is to collaborate during the ideation stage. When I first started designing, I felt like I needed to work privately and show others my work when it was complete. After working on this project, I learned that the best ideas can and often do come from collaborating with your colleagues. Additionally, getting the feedback and opinion of others, whether they be colleagues, users, or even friends, ultimately helps build a stronger product.

Project Outcome

Unfortunately, I was not able to see the finished result of this web application because the department I was working in dissolved before the baseball team was able to use the product. Even if our student team was able to complete this application, our client left the baseball team about a month after the department disbanded. I'm still grateful for the learning opportunity this project offered our team.
